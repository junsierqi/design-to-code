# Acceptance Report

Use this reference when preparing the final design-to-code evidence inside the `idea-to-code` closeout.

## Required Report Content

Include:

- restated user goal and design source
- source type and analysis method
- UI trace matrix with `C-*`, `I-*`, `S-*`, and optional `R-*` / `A-*` rows
- implementation summary with key files
- validation summary with validation types
- screenshots or artifact paths when generated
- known visual differences from the design
- mock backend/API contract and migration notes when mocks were used
- deferred items with concrete reasons and owners
- residual risks and follow-up recommendations

The durable lifecycle report is generated by `idea-to-code`; design-to-code adds UI-specific evidence to that lifecycle.

## Test Result Table

When tests are generated, each result should be reproducible:

| Test ID | Covers | Preconditions | Steps | Expected | Selector / Path | Assertion | Status | Artifact |
|---|---|---|---|---|---|---|---|---|
| TC-I-1 | I-1 | Dashboard loaded | Click Export | Toast appears | button[name=Export] | toast visible | pass | screenshots/export-toast.png |

Do not drop steps, selectors, assertions, or artifacts just to make the table shorter. If the table is too wide, split details into a second table rather than removing reproducibility data.

## Visual Acceptance

Separate functional correctness from visual acceptance:

- functional checks prove behavior
- screenshots support visual review
- known differences explain what changed from the design and why

Useful visual difference categories:

- typography
- spacing
- color
- border radius
- shadows
- icons/assets
- responsive layout
- motion

For each difference, state whether it is intentional, project-convention driven, backend/data driven, or deferred.

## Design Decisions

Include a compact decision table when implementation differs from the source design:

| Area | Design Evidence | Implementation Decision | Reason | Risk |
|---|---|---|---|---|
| spacing | Figma gap 18px | project spacing token 16px | aligns with system scale | minor visual mismatch |

Important decision categories:

- design token substitutions
- missing font or asset substitutions
- icon library replacements
- responsive behavior inferred from one frame
- motion omitted or simplified
- project component reused instead of custom visual clone
- accessibility changes that alter visual layout

## Accessibility And I18n Notes

Report:

- keyboard path tested or deferred
- accessible names for icon-only actions
- focus and modal behavior
- contrast concerns
- i18n system used or hardcoded string rationale
- long-text handling
- date, number, currency, and unit formatting decisions

Do not claim accessibility or localization completeness unless it was actually tested.

## Backend Handoff

If mocks were created, document:

- endpoint or service name
- method
- request shape
- response shape
- mock file path
- environment variable or switch to real API
- deferred integration tests

Make it possible for a backend developer to replace mocks without reading the whole implementation.

## Artifact Checklist

Before closeout, confirm the report references actual generated or inspected artifacts:

- design source path, URL, or screenshot
- implemented route or component path
- UI trace matrix
- validation command output
- screenshots or explicit reason screenshots were not produced
- visual difference table
- token/asset decision table when relevant
- accessibility/i18n notes when relevant
- deferred backend or design dependencies

## Closeout Language

The final chat response must follow the active `idea-to-code` closeout contract. Keep it short, but include design-to-code-specific details under validation and key technical details:

- UI trace coverage
- validation command and validation type
- screenshot/report paths when available
- unresolved visual or backend risks

Use `Progress` or `Blocked` instead of `Completed` if UI behavior, trace coverage, or validation evidence is missing.
